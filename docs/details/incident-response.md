# 障害対応フロー

## 対応フロー

1. **検知**
   - アクセスログ（`http_status >= 500` など）やアプリケーションのエラーログから障害を検知する。
   - 現時点では監視ツール（Datadog等）との連携は未実施のため、当面はログの目視・手動確認による検知を想定する。今後の連携準備として [datadog-monitors.md](datadog-monitors.md) にモニター定義を用意している。
2. **一次対応・切り分け**
   - `trace_id` / `request_id` をキーにログを追い、影響範囲（対象エンドポイント・対象ユーザー数）を確認する。
   - 緊急度が高い場合は暫定対応（該当機能の一時停止、ロールバック等）を優先する。
3. **復旧**
   - 原因箇所を修正し、動作確認後に本番反映する。
4. **ポストモーテム作成**
   - 復旧後、下記テンプレートを使ってポストモーテムを作成し、`docs/` 配下または社内の記録場所に保存する。

## ログコスト管理について（今後の課題）

- 現状はログレベルをINFOまでに制限し、高頻度ポーリングエンドポイント（例: `/api/posts/new-count`）のアクセスログを除外することでログ量を抑制している。
- Datadog等と連携する際は、ログのインデックス保持期間の設定やサンプリングによって追加のコスト調整余地がある。連携時に改めて検討する。
